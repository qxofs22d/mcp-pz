## Требования к оркестратору (системе обработки данных)

Оркестратор — это программный модуль (или набор скриптов), который выполняет все операции с JSON-файлами классов: импорт, валидацию, построение графа, трассировку, генерацию отчётов и взаимодействие с пользователем через CLI. Ниже перечислены ключевые функциональные требования, вытекающие из обсуждения.

### 1. Загрузка и парсинг данных
- Чтение всех JSON-файлов классов из заданной директории.
- Проверка синтаксической корректности JSON.
- Построение внутреннего представления (объектной модели) с индексацией по `id` (в пределах каждого класса и глобально, если требуется).

### 2. Валидация целостности графа
- Проверка, что все ссылки (`links`, `inputs`, `outputs`, `source_ref` и т.п.) указывают на существующие объекты (по `id` и классу).
- Выявление «висячих» ссылок (ссылок на несуществующие объекты).
- Проверка обязательных полей у объектов (согласно схеме каждого класса).
- Генерация отчёта об ошибках и предупреждениях.

### 3. Обнаружение циклов
- Анализ графа зависимостей на наличие циклов (например, метод А ссылается на метод Б, а Б — на А).
- Классификация циклов: допустимые (например, итерационные расчёты) и ошибочные (логические круги).
- Предоставление пользователю информации о циклах для ручного анализа.

### 4. Трассировка влияния (impact analysis)
- По заданному идентификатору объекта найти все объекты, которые прямо или косвенно зависят от него (потомки по графу).
- Построение списка объектов, которые будут затронуты при изменении исходных данных, метода или нормативного документа.
- Экспорт результатов трассировки в читаемый формат (Markdown, CSV).

### 5. Версионирование и история изменений
- Поддержка работы с системой контроля версий (Git) для хранения истории изменений JSON-файлов.
- Возможность указания версии объекта (поле `version` в классе или через коммит Git).
- При связывании объектов должна быть возможность ссылаться на конкретную версию (например, с помощью хэша коммита или тега).
- Механизм сравнения версий объектов и выявления изменений.

### 6. Автоматическая реклассификация (для класса 15)
- Периодический анализ объектов в классе 15 («Прочие данные»).
- Группировка объектов по структурным признакам (например, по наличию одинаковых полей или по семантической близости через эмбеддинги).
- Формирование предложений по созданию новых классов или перемещению объектов в существующие классы.
- Интерфейс для утверждения реклассификации пользователем.

### 7. Генерация отчётов и дайджестов
- Построение по запросу пользователя отчётов:
  - Список применённых нормативных документов с указанием пунктов.
  - Дерево зависимостей для конкретного результата или решения.
  - Сводка по проверкам (checks) из класса 11.
  - Сравнение вариантов из класса 12.
- Вывод результатов в формате Markdown (md-файлы) для удобного просмотра и документирования.
- Возможность генерации части текста пояснительной записки на основе структуры документа (класс 5) и связанных объектов, с использованием шаблонов.

### 8. Интеграция с векторным индексом
- Предоставление API для извлечения текстовых фрагментов по ссылкам `source_ref`.
- Возможность семантического поиска по текстам нормативных документов, отчётов и других источников.
- Поддержка хранения эмбеддингов и быстрого доступа к ним.

### 9. Производительность и масштабирование
- Для проектов с тысячами объектов обеспечить эффективное построение графа (кеширование, индексация).
- При необходимости поддержка выгрузки графа в графовую БД (например, Neo4j) для сложных аналитических запросов.
- Возможность инкрементальной обработки изменений (пересчёт только затронутых частей).

### 10. Интерфейс взаимодействия (CLI)
- Реализация командной строки с набором команд:
  - `validate` — проверка целостности.
  - `impact <id>` — трассировка влияния.
  - `report <type>` — генерация отчётов.
  - `graph` — визуализация графа (в виде текстового описания или экспорт в Graphviz).
  - `reclassify` — запуск анализа для класса 15.
  - `version` — управление версиями.
- Вывод результатов в человекочитаемом виде (Markdown, таблицы).

Эти требования должны быть реализованы в оркестраторе, чтобы система классов стала полноценным инструментом автоматизации проектной деятельности.
