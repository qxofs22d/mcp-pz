### **Общая цель**
Создать параметрическую систему, которая позволяет:
- Однократно разобрать эталонную пояснительную записку инженерного проекта (любого раздела: КЖ, ОВК и т.д.) вместе со всей связанной нормативной базой (ГОСТ, СП, МУ, ТСН и пр.).
- Извлечь из неё структуру, формулы, инструкции, показатели и текстовые описания, организовав их в систему взаимосвязанных классов данных.
- Затем для нового проекта, имея новые исходные данные, автоматически генерировать новую пояснительную записку с актуальными результатами расчётов и соответствующими текстовыми обоснованиями.

### **Ключевые принципы (параметризация)**
- **Формулы и алгоритмы занимают отдельное место** — они могут быть исполнены как скрипты и чётко отделены от исходных данных.
- **Ветвления в расчётах** учитываются: логика выбора метода в зависимости от промежуточных условий явно описывается.
- **Разделение данных**:
  - *Структурированные данные* (JSON по классам) — содержат всё необходимое для выполнения расчётов и получения результатов.
  - *Семантические данные* (векторный индекс полных текстов) — используются для связывания при разборке и для генерации текстовых пояснений.
- **Целостность и связность** — все объекты имеют уникальные идентификаторы и ссылки друг на друга, что позволяет строить граф зависимостей.
- **Версионирование** — поддерживается через внешнюю систему Git (с возможностью расширения).
- **Расширяемость** — предусмотрен класс «Прочие данные» для неклассифицированных элементов и механизм автоматической реклассификации.

### **Архитектура (на основе MCP-серверов)**
1. **MCP-сервер «Разборщик»** — выполняет первичный анализ эталонной ПЗ и всех связанных документов. Результат: JSON-наборы по 15 классам и векторный индекс полных текстов.
2. **MCP-сервер «Data Core»** — универсальный вычислитель без состояния. Получает от оркестратора фрагмент данных (формулу/инструкцию и значения переменных), превращает их в исполняемый код, выполняет и возвращает результат.
3. **MCP-сервер «Document Composer»** — отвечает за формирование итогового документа. Преобразует результаты расчётов и текстовые фрагменты (из векторного индекса) в LaTeX-подобный вид и собирает разделы согласно структуре документа (класс 5).
4. **Оркестратор** — центральный координатор с CLI-интерфейсом. Управляет всей последовательностью: запускает разборку, хранит наборы данных, выполняет валидацию, трассировку, генерацию отчётов, отправляет запросы в Data Core, передаёт результаты в Composer, обеспечивает диалог с пользователем. Все коммуникации между серверами идут только через оркестратор.

### **Этапы работы**

#### **I. Разборка (однократно)**
1. **Идентификация раздела и типа проекта** (например, КЖ → фундаменты).
2. **Определение необходимой нормативной базы** по ссылкам в ПЗ.
3. **Проверка наличия всех документов** в папке проекта; при отсутствии — запрос у пользователя и загрузка.
4. **Векторизация всех документов** (ПЗ, ГОСТы, СП, МУ и т.д.) — создание векторного индекса для семантического доступа.
5. **Извлечение структурированных данных с использованием векторного индекса**:
   - Разборщик идентифицирует в ПЗ все значимые элементы (формулы, инструкции, показатели, ссылки на нормативы) и с помощью семантического поиска находит соответствующие фрагменты в других документах.
   - Извлечённые данные раскладываются по 15 классам (JSON-файлы), каждый объект получает уникальный `id` и ссылки на связанные объекты и на фрагменты в векторном индексе.
   - Методы из МУ извлекаются в сокращённом виде, достаточном для расчётов; полные описания остаются в индексе.
6. **Сохранение JSON-наборов** (они содержат всё необходимое для вычислений, но не включают развёрнутые текстовые пояснения).

#### **II. Работа с новым проектом (многократно)**
1. **Подготовка исходных данных**:
   - Если есть отдельный документ с исходными данными — он загружается и проходит аналогичную векторизацию/извлечение (попадает в класс 3).
   - Если нет — набор исходных данных формируется на основе показателей из эталонной ПЗ (класс 3), с возможностью редактирования через оркестратор (CLI).
2. **Валидация и анализ** (оркестратор):
   - Проверка целостности графа (все ссылки валидны).
   - Обнаружение циклов зависимостей.
   - Трассировка влияния (опционально).
3. **Выполнение расчётов**:
   - Оркестратор для каждого расчётного блока отправляет в Data Core запрос, содержащий формулу/инструкцию (из класса 6 или 7) и текущие значения переменных (из классов 3, 4, 10, 11).
   - Data Core динамически компилирует скрипт, выполняет, возвращает результат оркестратору.
   - При ветвлениях процесс повторяется с выбором соответствующей ветки.
   - Результаты сохраняются в классы 10 (промежуточные) и 11 (итоговые).
4. **Формирование документа**:
   - Для каждого раздела (класс 5) оркестратор собирает связанные объекты (результаты расчётов, проектные решения, инструктивные указания и т.д.).
   - При необходимости из векторного индекса по сохранённым ссылкам извлекаются полные текстовые описания.
   - Document Composer форматирует данные вместе с текстами в LaTeX-подобный вид и собирает разделы.
   - Возможно формирование промежуточных дайджестов (Markdown) для анализа.
5. **Генерация отчётов и итоговой ПЗ**:
   - Оркестратор по запросу пользователя формирует отчёты (список нормативов, дерево зависимостей, сводка проверок и т.п.).
   - Финальная пояснительная записка сохраняется в нужном формате.

### **Роли данных**
- **JSON-наборы (15 классов)** — вычислительное ядро. Содержат минимум, необходимый для получения всех результатов и решений. Недостаточны для оформления текста.
- **Векторный индекс** — семантическая оболочка. Используется на этапе разборки для связывания данных (поиск соответствий в нормативных документах) и на этапе генерации для извлечения полных текстов пояснений.
