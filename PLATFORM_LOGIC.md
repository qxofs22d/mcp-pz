# Логика работы Платформы MCP-PZ

## Концепция

**Платформа MCP-PZ** — это набор MCP-серверов (инструментов) для автоматизации подготовки пояснительных записок в строительстве. Платформа отделена от данных проектов.

## Ключевое разделение

| Сущность | Путь | Назначение |
|----------|------|------------|
| **Платформа** | `d:\dev\mcp-pz\` | Код MCP-серверов, утилиты |
| **Рабочий проект** | `d:\Projects\{name}\` | Данные конкретного проекта ПЗ |

> **Важно:** Платформа — это инструмент, Рабочий проект — это данные. Не путать!

## Архитектура

### Платформа (`d:\dev\mcp-pz\`)

```
d:\dev\mcp-pz\
├── servers/                      # MCP-серверы
│   ├── logger/                   # логирование
│   ├── project_manager/          # управление проектами
│   ├── pdf_decomposer/           # декомпозиция PDF
│   ├── vector_storage/           # векторный поиск
│   └── data_core/                # хранилище данных
├── data/                         # данные платформы
│   └── registry.json             # реестр всех проектов
├── .clinerules                   # системный промпт для Cline
├── .gitignore                    # исключения для Git
├── ARCHITECTURE.md               # архитектура серверов
├── PLATFORM_LOGIC.md             # этот документ
├── CLINE_INTERFACE.md            # настройка интерфейса Cline
└── requirements.txt              # зависимости Python
```

### Реестр проектов

**Файл:** `data/registry.json`

**Назначение:** Централизованный список всех проектов ПЗ

**Структура:**
```json
{
  "version": "1.0",
  "projects": [
    {
      "id": "kr_foundation",
      "name": "Фундамент здания",
      "path": "d:\\Projects\\kr_foundation",
      "created": "2026-03-02T18:21:00",
      "status": "active"
    }
  ],
  "metadata": {
    "created": "2026-03-02T18:00:00",
    "last_modified": "2026-03-03T15:30:00",
    "description": "Реестр проектов ПЗ"
  }
}
```

**Использование:**
- Project Manager читает/пишет реестр
- Отслеживание всех проектов в одном месте
- Быстрый поиск проекта по ID или имени

## Workspace = Проект

### Принцип работы

VS Code открывается с **папкой проекта**, а не папкой платформы:

```
✅ Правильно:   VS Code → Open Folder → d:\Projects\kr_foundation
❌ Неправильно: VS Code → Open Folder → d:\dev\mcp-pz
```

### Преимущества

1. **`${workspaceFolder}`** автоматически указывает на проект
2. Каждый проект независим и содержит свою конфигурацию MCP
3. Можно открывать несколько проектов в разных окнах VS Code
4. Платформа — просто набор инструментов, не привязана к данным

## Конфигурация MCP

### Расположение

Конфигурация MCP находится **в проекте**, а не в платформе:

```
d:\Projects\kr_foundation\.vscode\mcp.json
```

### Структура

```json
{
  "mcpServers": {
    "logger": {
      "command": "D:\\dev\\.venvs\\pdf-analyzer\\Scripts\\python.exe",
      "args": ["-m", "mcp", "run", "d:\\dev\\mcp-pz\\servers\\logger\\server.py"],
      "env": {
        "PROJECT_PATH": "${workspaceFolder}"
      }
    },
    // ... другие серверы
  }
}
```

### Механизм PROJECT_PATH

1. VS Code открывает папку проекта как workspace
2. `${workspaceFolder}` = путь к проекту (например, `d:\Projects\kr_foundation`)
3. MCP-серверы получают `PROJECT_PATH` через переменную окружения
4. Серверы используют `PROJECT_PATH` для определения путей к данным:
   - `vector_index/` → векторный индекс
   - `data/` → расчётные данные
   - `pdf_cache/` → кэш PDF

## MCP-серверы

### 1. Logger (`servers/logger/`)
- **Назначение:** Логирование действий и системных сообщений
- **Инструменты:** `log` — запись логов
- **Типы записей:** `regular`, `system`

### 2. Project Manager (`servers/project_manager/`)
- **Назначение:** Управление проектами ПЗ
- **Инструменты:**
  - `init_project` — инициализация нового проекта
  - `detect_project` — определение типа проекта
  - `load_project` — загрузка существующего проекта
  - `get_project_info` — информация о проекте
  - `add_pdf` — добавление PDF в проект
  - `get_state` / `update_state` — работа с состоянием

### 3. PDF Decomposer (`servers/pdf_decomposer/`)
- **Назначение:** Анализ и декомпозиция PDF-файлов
- **Инструменты:**
  - `decompose_pdf` — полная декомпозиция PDF
  - `extract_tables` — извлечение таблиц
  - `extract_formulas` — поиск формул
  - `get_structure` — иерархия разделов

### 4. Vector Storage (`servers/vector_storage/`)
- **Назначение:** Семантический поиск по индексированным данным
- **Инструменты:**
  - `add_document_chunks` — добавление чанков
  - `search` — поиск по коллекции
  - `list_collections` — список коллекций
  - `delete_document` — удаление документа
  - `split_text` — разбивка текста на чанки
- **Коллекции:** `documents`, `pdf`

### 5. Data Core (`servers/data_core/`)
- **Назначение:** Хранилище расчётных данных, параметров и формул
- **Инструменты:**
  - `get_parameter` / `set_parameter` — работа с параметрами
  - `add_formula` / `execute_formula` — работа с формулами
  - `run_script` — запуск скриптов
  - `export_data` / `import_data` — экспорт/импорт данных

## Порядок работы над проектом ПЗ

### Этап 1: Создание и инициализация проекта

```
1. Создать папку: d:\Projects\{project_name}\
2. Создать .vscode\mcp.json (скопировать из существующего проекта)
3. Создать .clinerules с описанием проекта (дисциплина, нормативы)
4. Открыть папку в VS Code: File → Open Folder
5. Вызвать project_manager: init_project
```

### Этап 2: Анализ эталонной ПЗ и сбор документов

```
1. Пользователь помещает эталонную ПЗ в папку pdf/
2. Вызвать pdf_decomposer: decompose_pdf для анализа ПЗ
3. Извлечь ссылки на нормативные документы (СП, ГОСТ, МУ, ТСН)
4. Сформировать список необходимых документов

5. ПРОВЕРКА НАЛИЧИЯ ДОКУМЕНТОВ:
   • Проверить pdf/ на наличие всех документов
   • Вывести список недостающих документов
   • Попросить пользователя загрузить недостающие

6. Повторять проверку пока все документы не будут в наличии
```

### Этап 3: Векторизация (только после сбора всех документов)

```
1. Для каждого PDF в pdf/:
   • Извлечь текст
   • Разбить на чанки (vector_storage: split_text)
   • Добавить в коллекцию (vector_storage: add_document_chunks)
2. Создать единый векторный индекс для семантического поиска
```

### Этап 4: Декомпозиция и классификация данных

```
1. Детальный анализ эталонной ПЗ с использованием векторного индекса
2. Извлечение:
   • Структуры документа → document_structure.json
   • Исходных данных → input_data.json
   • Формул → formulas.json
   • Методов расчёта → methods.json
   • Результатов → final_results.json
   • Допущений → assumptions.json
   • Проектных решений → design_decisions.json
   • Ссылок на нормативы → regulatory_framework.json
   • Прочих данных → misc.json
3. Валидация связей между объектами (validate_data)
4. Построение графа зависимостей (get_graph_stats)
```

### Этап 5: Работа с новым проектом

```
1. Загрузить новые исходные данные (input_data)
2. Валидация графа зависимостей
3. Выполнение расчётов (data_core)
4. Генерация новой ПЗ (document_composer)
```

## Ограничения

### Cline CLI

`${workspaceFolder}` — это переменная VS Code, не Cline CLI.

Для работы через CLI требуется:
- Установить `PROJECT_PATH` вручную перед запуском
- Или доработать серверы для определения текущей директории

## Зависимости

- `mcp` — MCP SDK
- `PyMuPDF`, `pdfplumber`, `pandas` — работа с PDF
- `sentence-transformers`, `numpy` — векторный поиск
- `asteval` — вычисление формул
- `aiohttp` — асинхронные HTTP-запросы
- `python-dotenv` — загрузка конфигурации

## Резюме

1. **Платформа** (`mcp-pz`) — содержит только код серверов
2. **Проект** (`d:\Projects\{name}`) — содержит данные и конфигурацию MCP
3. **Workspace = Проект** — VS Code открывает папку проекта
4. **`${workspaceFolder}`** — автоматически определяет путь к проекту
5. **Независимость** — каждый проект имеет свою конфигурацию MCP
